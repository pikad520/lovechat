# Feature Specification: LoveChat — AI 角色对话 macOS 应用

**Feature Branch**: `001-lovechat-app`

**Created**: 2026-06-10

**Status**: Draft

**Input**: User description: "基于 DESIGN.md 构建带界面的 AI 角色对话 macOS 应用：可配置多个 AI Chat/Imagine Provider，创建角色卡片，与角色进行流式对话，支持心理活动展示、图片生成联动、上下文记忆与压缩、对话管理与导出。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 配置服务并完成一次流式对话 (Priority: P1)

用户首次打开应用，添加一个 AI Chat Provider（填写名称、服务地址、密钥、模型名、协议类型），创建一个最简角色（仅必填字段），进入对话界面发送一条消息，看到角色的回复以流式方式逐字出现。

**Why this priority**: 这是应用存在的根本价值——没有「配置 + 对话」闭环，其他一切功能都无从谈起。

**Independent Test**: 全新安装后，仅通过「添加 Provider → 创建角色 → 发送一条消息 → 收到流式回复」即可完整验证，交付可用的最小产品。

**Acceptance Scenarios**:

1. **Given** 全新安装、无任何配置，**When** 用户添加一个 Chat Provider 并保存，**Then** Provider 出现在列表中，且密钥被存入系统钥匙串（任何本地数据文件中不出现明文密钥）。
2. **Given** 已配置 Provider，**When** 用户对其执行连通性测试，**Then** 应用明确反馈成功或失败（含可理解的失败原因）。
3. **Given** 已有 Provider 与角色，**When** 用户发送一条消息，**Then** 回复以流式逐字出现，期间用户可随时点击「停止生成」。
4. **Given** 回复正在流式输出，**When** 网络中断，**Then** 对话流中显示友好错误文案并提供重试，已输出的部分内容不丢失。

---

### User Story 2 - 创建并定制角色卡片 (Priority: P2)

用户创建多个角色，为每个角色填写名称、性别、外貌、灵魂（性格内核）、对用户的称呼、说话风格及自由补充设定；可为角色开启「展示心理活动」开关，开启后角色回复中括号包裹的心理活动以特殊样式渲染。

**Why this priority**: 角色是产品的差异化核心；P1 已可用最简角色对话，本故事补全角色深度定制。

**Independent Test**: 创建两个设定迥异的角色分别对话，验证回复风格、称呼随角色设定变化；开关心理活动并观察渲染差异。

**Acceptance Scenarios**:

1. **Given** 角色创建界面，**When** 用户填写全部字段并保存，**Then** 角色出现在角色列表，可被选择用于新对话。
2. **Given** 角色开启「展示心理活动」，**When** 角色回复中包含小括号包裹的内容，**Then** 该内容以区别于正文的特殊样式渲染。
3. **Given** 角色开启「展示心理活动」，**When** 模型回复未遵守括号格式或括号不配对，**Then** 内容按普通文本容错显示，不报错、不丢字。
4. **Given** 角色关闭「展示心理活动」，**When** 角色回复，**Then** 系统提示词中不包含心理活动指令。

---

### User Story 3 - 管理对话历史 (Priority: P3)

用户在卡片式历史列表中查看所有对话，可选择任意历史对话继续聊、删除对话、新开对话；对话内可停止生成、重新生成最后一条回复、编辑消息、删除消息、把对话导出为 Markdown。

**Why this priority**: 持久化与可管理性让应用从「一次性玩具」变成日常工具，但依赖 P1 的对话能力。

**Independent Test**: 创建多个对话后重启应用，验证历史完整保留；逐一执行继续/删除/导出/编辑/重新生成操作。

**Acceptance Scenarios**:

1. **Given** 已进行过多个对话，**When** 用户重启应用，**Then** 历史对话以卡片列表完整展示。
2. **Given** 历史对话列表，**When** 用户选择某对话，**Then** 进入该对话并可继续发消息，上下文延续。
3. **Given** 一段对话，**When** 用户导出为 Markdown，**Then** 导出文件包含完整对话内容，且不含任何密钥或服务配置信息。
4. **Given** 最后一条回复不满意，**When** 用户点击「重新生成」，**Then** 旧回复被替换为新的流式回复。
5. **Given** 任一消息，**When** 用户编辑或删除它，**Then** 变更立即持久化并反映在后续上下文中。

---

### User Story 4 - 上下文记忆与自动压缩 (Priority: P4)

用户可设置每个对话记住的轮数（1–20，默认 10）。对话超过阈值后，滑出记忆窗口的旧消息被后台压缩为摘要并注入系统提示，使角色「记得」久远内容而不超出上下文限制。

**Why this priority**: 长对话体验的质量保障，依赖 P1/P3 的对话与持久化。

**Independent Test**: 把记忆轮数设为最小值，进行超过阈值的多轮对话，验证旧内容以摘要形式仍被角色引用；人为使压缩失败验证降级为截断。

**Acceptance Scenarios**:

1. **Given** 记忆轮数设置界面，**When** 用户尝试输入超出 1–20 的值，**Then** 界面阻止并约束在合法范围。
2. **Given** 压缩阈值与记忆轮数设置，**When** 用户尝试将压缩阈值设为大于记忆轮数，**Then** 界面强制阻止。
3. **Given** 对话超过记忆轮数，**When** 旧消息滑出窗口，**Then** 压缩在后台进行，用户发消息不被阻塞。
4. **Given** 压缩调用失败，**When** 旧消息需要被处理,**Then** 系统静默降级为直接截断旧消息，对话不中断。

---

### User Story 5 - 图片生成联动 (Priority: P5)

用户配置 AI Imagine Provider 并为角色开启「允许发送图片」。此后每条用户消息会先经一次快速判断；当对话情境适合时，角色会基于其外貌设定与当前情境生成一张图片发给用户，图片立即保存到本地。

**Why this priority**: 增强沉浸感的加分项，依赖 P1/P2 的对话与角色体系。

**Independent Test**: 配置 Imagine Provider、开启角色开关，用明显「求图」的消息触发生成；断开 Imagine 服务验证文字回复不受影响。

**Acceptance Scenarios**:

1. **Given** 角色开启「允许发送图片」，**When** 用户发送消息，**Then** 系统先用一次非流式短调用判断是否生图，再正常进行流式文字回复。
2. **Given** 判断结果为「应生图」，**When** 图片生成成功，**Then** 图片出现在对话流中且文件已保存到本地（重启后仍可见）。
3. **Given** 图片生成失败（任何原因），**When** 文字回复正常，**Then** 文字回复完整展示，仅图片部分以友好方式提示失败。
4. **Given** 角色关闭「允许发送图片」，**When** 用户发送任何消息，**Then** 不发生生图判断调用。

---

### User Story 6 - 思考模式 (Priority: P6)

用户可在 Provider 配置中开启思考模式，让支持的模型进行更深入的推理后再回复；模型不支持时静默降级，对话照常进行。

**Why this priority**: 锦上添花的质量选项，且要求对不支持的模型零干扰。

**Independent Test**: 分别对支持与不支持思考模式的模型开启该开关对话，前者生效、后者无感降级，均不报错。

**Acceptance Scenarios**:

1. **Given** Provider 开启思考模式且模型支持，**When** 用户发消息，**Then** 请求携带对应思考参数且回复正常。
2. **Given** Provider 开启思考模式但模型不支持，**When** 用户发消息，**Then** 对话正常完成，无错误提示、无中断。

---

### Edge Cases

- 服务方因内容审核拒绝回复 → 对话流中显示友好文案，可重试。
- 限流（429）或服务端错误 → 友好文案 + 重试，不崩溃。
- 流式输出中途中断 → 已收内容保留，可恢复或重新生成。
- 心理活动括号嵌套、不闭合、中英文括号混用 → 容错按普通文本显示。
- 记忆轮数边界值 1 和 20 → 均可正常工作。
- 删除正在对话使用的角色或 Provider → 已有对话历史仍可浏览与导出，但继续发消息时给出明确提示。
- 同时进行的后台压缩与用户新消息 → 互不阻塞，最终上下文一致。
- 生成图片的下载链接过期前未完成下载 → 视为生图失败，按失败路径处理。

## Requirements *(mandatory)*

### Functional Requirements

#### Provider 管理

- **FR-001**: 系统 MUST 支持配置多个 AI Chat Provider，字段含：名称、Base URL、API Key、模型名、协议类型（OpenAI 标准 / Anthropic 标准）。
- **FR-002**: 系统 MUST 支持配置多个 AI Imagine Provider，字段含：名称、Base URL、API Key、模型名（优先 OpenAI 标准 Images API）。
- **FR-003**: 所有 API Key MUST 仅存储于 macOS 钥匙串，任何本地数据文件、日志、导出内容中 MUST NOT 出现明文密钥。
- **FR-004**: 系统 MUST 为 Chat Provider 提供连通性测试，并反馈成功/失败及原因。
- **FR-005**: Chat Provider MUST 支持思考模式开关（Anthropic → extended thinking；OpenAI → reasoning effort）；模型不支持时 MUST 静默降级，不报错、不中断。

#### 对话能力

- **FR-006**: 聊天回复 MUST 以流式（SSE）方式逐字呈现。
- **FR-007**: 生图判断与上下文压缩这两类内部调用 MUST 为非流式，且限制输出长度。
- **FR-008**: 用户 MUST 能在生成期间停止生成。
- **FR-009**: 用户 MUST 能重新生成最后一条回复、编辑消息、删除消息。
- **FR-010**: API 错误（审核拒绝、网络错误、限流等）MUST 以友好文案展示在对话流中并支持重试；流式中断 MUST 可恢复或重新生成。

#### 角色卡片

- **FR-011**: 用户 MUST 能创建多个角色，字段：名称、性别、外貌、灵魂（性格内核）、对用户的称呼、说话风格、自由补充设定。
- **FR-012**: 角色 MUST 有「展示心理活动」开关；开启时心理活动以小括号包裹并由界面渲染为特殊样式；解析失败时 MUST 按普通文本容错显示。
- **FR-013**: 角色 MUST 有「允许发送图片」开关；开启时每条用户消息先经一次生图判断调用，判断为是则调用 Imagine Provider 生成一张图。
- **FR-014**: 心理活动提示词、生图判断提示词、生图提示词模板 MUST 硬编码于程序内，用户与角色 MUST NOT 能自定义。

#### 上下文与记忆

- **FR-015**: 记忆对话轮数 MUST 可在 1–20 范围内自定义，默认 10；界面 MUST 拒绝范围外输入。
- **FR-016**: 对话超过阈值时，滑出记忆窗口的旧消息 MUST 由同一 Chat Provider 压缩为摘要并注入系统提示；压缩阈值 MUST ≤ 记忆轮数（界面层强制）。
- **FR-017**: 压缩 MUST 后台异步执行、不阻塞发消息；压缩失败 MUST 降级为直接截断旧消息。
- **FR-018**: 系统提示词组成 MUST 为：角色设定 + 心理活动提示词（若开启）+ 压缩后的历史摘要（若有）。

#### 持久化与对话管理

- **FR-019**: 对话历史 MUST 本地持久化，应用重启后完整可见。
- **FR-020**: 历史对话 MUST 以卡片式列表展示，可选择任意对话继续、删除对话、新开对话。
- **FR-021**: 对话 MUST 可导出为 Markdown，导出内容 MUST NOT 含密钥或服务配置。
- **FR-022**: 生成的图片 MUST 立即下载保存到本地存储（远端链接短期过期，不可仅保存链接）。

### Key Entities

- **Chat Provider**: 一个可对话的 AI 服务配置；属性：名称、服务地址、密钥（钥匙串引用）、模型名、协议类型、思考模式开关。
- **Imagine Provider**: 一个可生图的 AI 服务配置；属性：名称、服务地址、密钥（钥匙串引用）、模型名。
- **角色（Character）**: 对话人格；属性：名称、性别、外貌、灵魂、称呼、说话风格、补充设定、心理活动开关、允许发图开关。
- **对话（Conversation）**: 一次持续聊天；关联一个角色与一个 Chat Provider（可选关联 Imagine Provider）；属性：标题、创建/更新时间、记忆轮数、历史摘要。
- **消息（Message）**: 对话中的一条内容；属性：角色方（用户/角色）、文本、可选图片、时间、状态（生成中/完成/失败）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户从首次打开到收到第一条角色回复，全程不超过 3 分钟（含配置 Provider 与创建角色）。
- **SC-002**: 流式回复首字出现延迟与直接调用该 AI 服务相比无可感知差异；输出期间界面保持可交互（可停止、可滚动）。
- **SC-003**: 100% 的 API 错误场景（审核、网络、限流）均以友好文案展示且可重试，零未处理崩溃。
- **SC-004**: 在对话超过记忆阈值后，角色仍能正确引用早期关键信息（通过摘要），且用户发消息从未因压缩被阻塞。
- **SC-005**: 任何本地文件（数据库、日志、导出）中均检索不到明文 API Key。
- **SC-006**: 生图失败场景下，文字回复 100% 正常展示。

## Assumptions

- 单用户单机使用，无账号体系、无云同步;数据全部本地。
- 用户自备各 AI 服务的有效密钥与网络可达性。
- 「一轮」指一次用户消息 + 一次角色回复。
- 生图判断与压缩复用当前对话的 Chat Provider，不单独配置。
- 每次触发生图最多生成一张图片。
- 对话导出格式仅 Markdown（v1 范围）。
- 角色与 Provider 的删除采取保护历史的策略：历史对话可读，继续对话需重新关联。
